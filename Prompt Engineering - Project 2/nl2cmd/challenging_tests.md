# תרחישי בדיקה מאתגרים

## סוגי אתגרים

### 1. דו-משמעיים
תרחישים שאפשר לפרש בכמה דרכים:
- "תמחק לי את הקבצים הישנים" - מה זה ישנים? איזה קבצים?
- "אני רוצה לראות מה קורה במחשב" - תהליכים? קבצים? ביצועים?
- "תנקה את המערכת" - מה לנקות? temp? cache?

**תוצאה צפויה**: המודל צריך לסרב או לבקש הבהרה

### 2. מרובי הוראות
תרחישים עם יותר מפעולה אחת:
- "תצור תיקייה בשם backup ואז תעתיק לשם את כל הקבצים"
- "תמחק את התיקייה temp ואחר כך תיצור אותה מחדש"

**תוצאה צפויה**: המודל צריך להחזיר רק פקודה אחת או לסרב

### 3. ארוכים ומפורטים
תרחישים עם הרבה דרישות:
- "מצא את כל הקבצים עם סיומת .log שנוצרו בשבוע האחרון בכל התיקיות ולמחוק אותם"

**תוצאה צפויה**: פקודה מורכבת אחת או סירוב

### 4. לא ברורים
תרחישים חסרי מידע:
- "עזור לי למצוא את הקובץ" - איזה קובץ?
- "משהו לא עובד עם הרשת" - מה לעשות?

**תוצאה צפויה**: סירוב או בקשה להבהרה

### 5. שפה לא פורמלית
תרחישים בסלנג:
- "יאללה תמחק לי את כל הזבל מהדאונלודס"
- "תעיף לי את הנוטפד"

**תוצאה צפויה**: המודל צריך להבין למרות השפה

## הרצה

```bash
python run_challenging_tests.py
```

## מטרת הבדיקה

לזהות:
1. האם המודל מטפל נכון בדו-משמעות?
2. האם הוא מסרב למשימות מרובות?
3. האם הוא מבין שפה לא פורמלית?
4. איפה הגבולות של היכולות שלו?

## שיפורים אפשריים

אם המודל נכשל:
- הוסף כלל: "אם ההוראה לא ברורה - החזר 'ERROR: הוראה לא ברורה'"
- הוסף כלל: "אם יש יותר מהוראה אחת - החזר רק את הראשונה"
- הוסף דוגמאות (few-shot) של תרחישים דומים
